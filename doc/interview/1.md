## Java

### HashMap 原理

标准答案： 
- 底层是 数组 + 链表 + 红黑树（JDK1.8）
- key 经过 hash 后定位到数组 index
- 冲突时采用链表存储
- 当链表长度 ≥ 8 且数组容量 ≥ 64 时转为红黑树

扩展点：
- 扩容机制：容量翻倍，重新计算 index（rehash）



### 线程池

标准答案：
ThreadPoolExecutor 核心参数：
- corePoolSize：核心线程数
- maximumPoolSize：最大线程数
- workQueue：任务队列
- keepAliveTime：非核心线程存活时间
- handler：拒绝策略


执行流程：
- 先用核心线程
- 核心满 → 入队列
- 队列满 → 创建非核心线程
- 还满 → 拒绝策略


### JVM 内存结构

标准答案：
- 堆（Heap）：对象实例
- 栈（Stack）：方法调用
- 方法区（Metaspace）：类信息
- 程序计数器



## Spark 

### 转换算子和行动算子

标准答案：
- transformation：返回新的 RDD（懒执行）
- action：触发计算（如 count、collect）


### 宽依赖 vs 窄依赖

标准答案：
- 窄依赖：父分区 → 子分区一对一（map）
- 宽依赖：父分区 → 多个子分区（shuffle，如 groupBy）


### Spark 基础架构 & 执行流程

标准答案：
- Driver、Executor、Cluster Manager
    - Driver 作为主控节点，负责解析程序、构建DAG、划分Stage
    - Executor 作为计算节点，负责执行具体 Task 并管理数据缓存
    - Cluster Manager 负责资源管理和分配，如 Yarn、K8s等

标准答案：
- 提交 job
- 生成 DAG
- 划分 stage（根据 shuffle）
- 每个 stage 拆成 task
- executor 执行


### Spark 性能优化

- Shuffle 优化 (广播变量、分区调优)
- 内存管理（Execution / Storage Memory）
    - Spark 的内存管理主要负责在任务执行的过程中，对各类数据，如 Shuffle 数据、缓存数据、中间结果等 进行高效的分配和回收。特点：高性能 + 低 GC 开销
    - Spark 的内存总体可以分为两大类，执行内存和存储内存，剩下的就是其他内存
        - 执行内存：用于 Shuffle、Join、Sort、Agg 等计算过程中产生的中间数据
        - 存储内存：用户缓存 RDD、广播变量、序列化数据等
        - 其他内存：用户的代码、系统开销、元数据等
    - Spark 现在使用统一内存管理机制，在统一模型中，执行内存和存储内存共享同一块内存区域，执行内存可以动态借用存储内存，反向则不行。同时 Spark 还支持堆外内存来减少垃圾回收的开销。

### 数据倾斜

标准答案：
原因：
- key 分布不均

解决：
- 加盐（salting）
- map-side 聚合
- broadcast join
- 分区优化



### 任务变慢排查

标准答案：
排查步骤：
- Spark UI 看 stage/task
- 看是否 shuffle
- 看数据量变化
- 看 GC
- 看数据倾斜
